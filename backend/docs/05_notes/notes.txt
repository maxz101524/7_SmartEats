1. The Size vs. Quality Trade-off: Based on your 15-model test from the Hugging Face leaderboard, which size category provided the "sweet spot" for your project? Was a massive model significantly better than a smaller model for your specific task? Also, what would be expected cost of each of these pre-trained model to run on your server based on computation requirements.

we evaluated 15 models from the Hugging Face Open LLM Leaderboard, divided into five parameter-size categories: Ultra-Light (<1B), Small (1B–3B), Medium (3B–5B), Large (5B–7B), and Extra-Large (>7B). Each category contained three models, and we tested them using the same set of prompts related to our application task. The prompts were designed to measure several criteria including instruction adherence, output structure reliability, response quality, and inference speed. The goal was to identify the most practical model size for integration into our Django application. Based on these tests, the Small category (1B–3B parameters) provided the best overall balance between performance, reliability, and computational efficiency. Models in this range consistently followed the prompt instructions, produced structured outputs with fewer formatting errors, and generated responses quickly enough to support real-time interaction within a web application. Importantly, these models did not require excessive GPU resources and could be run locally without needing extremely expensive infrastructure. The Ultra-Light models (<1B parameters) were the fastest and easiest to run locally, but their performance was noticeably less reliable. Some responses were inconsistent, occasionally ignored parts of the prompt instructions, or produced incomplete structured outputs. For simple summarization tasks they were acceptable, but for our application—which requires predictable formatting and accurate responses—their reliability was not strong enough.

On the other hand, the Medium and Large models (3B–10B parameters) demonstrated slightly better reasoning ability and produced somewhat more detailed outputs. However, the improvement was not dramatic for our specific use case. Because our application focuses on structured responses rather than complex reasoning, the additional parameters did not translate into significantly better results. Instead, the larger models introduced slower inference times and much higher computational requirements. In practice, this means that massive models were not significantly better than smaller models for our task. The increased size improved some aspects of generation quality, but the difference was small compared to the additional hardware costs and latency. When considering deployment costs, model size has a major impact on server requirements. Ultra-Light models can often run on a CPU-only environment or a small GPU instance, which might cost approximately $50–$200 per month depending on the cloud provider. Small models (1B–3B) typically require a single GPU with around 8–16GB of VRAM, which corresponds to an estimated $400–$700 per month on common cloud GPU instances. Medium models (3B–7B) generally need more powerful GPUs with larger VRAM capacity, such as 24GB GPUs, increasing server costs to approximately $700–$1,200 per month. Large models in the 7B–10B range often require either high-memory GPUs or model quantization techniques, pushing costs to roughly $1,200–$2,000 per month. Extra-large models can require multiple GPUs, making them significantly more expensive and impractical for small-scale deployments. Considering both quality and infrastructure requirements, the 1B–3B parameter range clearly represented the “sweet spot” for our project. These models were capable enough to produce reliable outputs while still remaining affordable and efficient for local deployment within a Django application.




2. Cost & Scaling: If your application suddenly scaled to 10,000 daily users, would you stick with your local Django integration or move entirely to the API? Compare the potential server compute costs required to run your local model versus the per-token costs of an API.

If the application scaled to 10,000 daily users, relying entirely on a local Django-integrated model would become more challenging. Running a local Hugging Face model requires dedicated computing resources, typically a GPU-enabled server. For a model in the 1B–3B parameter range, a single cloud GPU instance might cost roughly $400–$700 per month. However, this assumes moderate traffic and limited concurrency. If many users send requests at the same time, multiple inference servers or load balancing would likely be required, which could increase monthly costs to $1,500–$3,000 or more depending on infrastructure and scaling needs. In contrast, using an external LLM API charges based on token usage rather than fixed hardware costs. Suppose the application receives 10,000 requests per day, with each request containing about 250 input tokens and generating 150 output tokens. Over a month, this equals approximately 75 million input tokens and 45 million output tokens. If an API costs around $0.10 per million input tokens and $0.40 per million output tokens, the estimated monthly cost would be: Input: 75M tokens × $0.10 / 1M ≈ $7.50 Output: 45M tokens × $0.40 / 1M ≈ $18.00 This results in a total of roughly $25–$30 per month, which is significantly cheaper than maintaining GPU servers for local inference. Even if token usage increased, the API approach would still likely remain more cost-efficient for most workloads. Because of this difference, if the application scaled to 10,000 daily users, it would make sense to shift most inference to an external API. APIs provide automatic scaling, lower infrastructure management, and predictable usage-based pricing. A practical solution would be a hybrid architecture, where the local Hugging Face model handles simple preprocessing or filtering tasks while the external API performs the final generation. This approach reduces API usage costs while still benefiting from scalable cloud-based inference.



3. Hybrid Potential: How could your local Hugging Face model and your external API work together to minimize costs while maintaining high quality?

The local Hugging Face model handles structured tasks (calorie estimation), while the Gemini API is used for complex conversational reasoning (meal recommendation).

Based on our testing, locally deployed models are sufficient to provide accurate and consistent responses within a reasonable inference time (~200 ms per request locally on GPU), while keeping infrastructure costs low. Running this model on our server configuration (16-core CPU, GPU with adequate VRAM, 64 GB RAM) supports real-time interaction in a Django application without requiring multiple high-end GPUs or additional cloud compute resources. Under these conditions—structured prompts, predictable outputs, and moderate reasoning—using a local pre-trained model is practical and cost-efficient.

However, routing tasks to the external API becomes preferable when user interactions involve open-ended conversations, personalized dietary advice, or context-sensitive menu recommendations. These tasks require stronger reasoning, natural language understanding, and broader knowledge than the small local models can reliably provide. While larger local models (7B–10B parameters or above) could improve output quality for these complex tasks, they would introduce major computational requirements, slower inference, and higher infrastructure costs. External APIs handle such complex queries efficiently, provide high-quality responses, and eliminate the need to maintain additional GPU infrastructure for rarely occurring high-complexity requests.

Therefore, our hybrid approach provides the best balance between cost efficiency and response quality.




4. Cost Comparison: Suppose you deploy a pre-trained model on Django, and you create a server of specific configurations (e.g., 16 core CPU, 16 core GPU, 64 GB RAM, 1 TB hard drive). Under what conditions it would be acceptable to use the pre-trained models vs when it would be better to switch to a paid API of comparable model size.

Assuming the current scale is 10,000 daily users, a single model run for calorie estimation takes approximately 2 seconds per user.
Running the 1.6B StableLM model locally on a single GPU deployment (8–16GB VRAM) typically costs around $400–$700 per month, depending on the cloud provider and usage.
From a computation perspective, if the model can process multiple requests concurrently:
a. If the model runs once per day per user, the total wall-clock processing time for 10,000 daily users would be about 167 hours per day.
b. If the model runs once per month per user, assuming each user changes their data/goal an average of 5 times, the total processing time would be about 5.56 hours per month.

For comparison, using a paid API with an average of 300 tokens per request at a cost of $0.03 per 1,000 tokens would result in:
a. 90,000,000 tokens and $2,700 per month if each user changes their data every day.
b. 3,000,000 tokens and $90 per month if each user changes their data once per month.

Thus, for workloads that are frequent and predictable, deploying a pre-trained model locally is generally more cost-effective. It also provides the advantages of consistent latency and independence from external services. In contrast, paid APIs are better suited for infrequent or unpredictable workloads, or for tasks that require reasoning or capabilities beyond what a Small/Medium local model can handle.

Week 5 — A5: Authentication, Google OAuth, Public API & Individual Deployment
================================================================================

SECTION 1: Token-Based Authentication (dj-rest-auth + DRF)
------------------------------------------------------------
- Used DRF's TokenAuthentication as the default auth class alongside
  SessionAuthentication (configured in REST_FRAMEWORK in base.py).
- RegisterAPIView (APIView) creates a Django User and linked UserProfile in one
  request; the UserProfile is auto-created via a post_save signal on User.
  Returns a token immediately so the user is logged in after registration.
- LoginAPIView authenticates with Django's authenticate(username, password),
  retrieves or creates a Token, and returns token + first_name + netID + goal
  so the frontend can personalize the UI without a second API call.
- Logout is frontend-only: localStorage.removeItem("authToken") clears the
  token; no server-side token invalidation (acceptable for this scope).
- Protected views use permission_classes = [IsAuthenticated]: UserProfileView
  (/api/profile/) and MealReportsView (/api/meal-reports/).
- Token is passed as "Authorization: Token <key>" header; frontend sets this
  via axios default headers after login.


SECTION 2: Google OAuth (django-allauth + @react-oauth/google)
---------------------------------------------------------------
- Backend: django-allauth with allauth.socialaccount.providers.google installed.
  GoogleLogin view (SocialLoginView from dj_rest_auth) uses GoogleOAuth2Adapter
  and OAuth2Client. Receives the Google id_token from the frontend and exchanges
  it for a DRF Token, returning res.data.key.
- SITE_ID = 2 required in base.py for allauth's Sites framework.
- SOCIALACCOUNT_PROVIDERS configured in base.py with profile + email scopes.
- Frontend: @react-oauth/google provides <GoogleLogin> component. On success,
  sends credentialResponse.credential (the id_token) to /api/google-login/.
  Stores the returned key as "authToken" in localStorage and navigates to /dishes.
- GoogleOAuthProvider wraps the app in main.tsx with the Google client ID from
  the environment.
- callback_url in GoogleLogin view is set to the local dev URL; production OAuth
  redirect URIs must be registered in Google Cloud Console for each deployed domain.


SECTION 3: Frontend Auth Flow
------------------------------
- Login.tsx: POST /api/login/ with netID + password. Stores token and firstName
  in localStorage. Navigates to /dishes on success.
- Register.tsx: POST /api/register/ with full profile data (netID, email,
  password, name, sex, age, height_cm, weight_kg, goal). Auto-logs in on success.
- GGLogin.tsx: Wraps @react-oauth/google's <GoogleLogin>. Reused inside Login.tsx
  so both login and signup surfaces offer "Continue with Google".
- logout.tsx: Removes authToken and userFirstName from localStorage, navigates
  to /login.
- Navbar.tsx: Reads localStorage.getItem("authToken") on each render. Shows
  Login button when no token, Logout button when token present.


SECTION 4: Public API for Vega-Lite (Part 3)

Week 4 — A4: Internal/External APIs, Charts, Exports & Deployment
================================================================================

SECTION 1: Internal JSON APIs for Charts
-----------------------------------------
- We continued to expose aggregation data as clean JSON from real Django models
  (e.g., dish and meal statistics) so it is "chart-ready" for Vega-Lite: each
  object already contains the x-axis key and metric field(s) (like date,
  dining_hall, category, or meal count).
- These JSON endpoints only return serialized data (no HTML), using
  JsonResponse(..., safe=False) or dict-shaped responses, which makes them usable
  both by our React frontend and by external tools like the Vega-Lite editor.
- In production, these APIs are reachable at /api/... URLs on Render, so
  classmates can point Vega-Lite's data.url directly at our deployed endpoints.


SECTION 2: External API Integration (Wger)
-------------------------------------------
- Implemented nutrition lookup at the view function `nutrition_lookup_view`
  using a keyless public API: Wger's ingredient endpoint
  (https://wger.de/api/v2/ingredient/).
- The view accepts a `?q=` query parameter, calls http_requests.get() with
  params (name, language, format, limit), sets timeout=5, and uses
  .raise_for_status() inside try/except blocks to handle timeouts and
  RequestException cleanly.
- The JSON body from Wger is normalized into a compact list of external_items
  with per_100g macros (kcal, protein, carbohydrates, fat, fiber, sodium).
- In parallel, we query our own Dish model with dish_name__icontains=query and
  join related dining hall information; this gives us internal_matches that we
  do NOT persist back into the database.
- The response combines both sources plus optional analytics per user:
  if a netID is provided, we look up UserProfile, compute BMR/TDEE using
  Mifflin-St Jeor, derive daily macro targets based on goal (fat_loss or
  muscle_gain), compare against today's Meal intake, and return remaining macros.
- Final output is a single JsonResponse with keys:
  query, external_results, internal_matches, and user_analysis, demonstrating
  triangulation between external and internal data without storing the external
  API payload.


SECTION 3: CSV + JSON Export of Meals
--------------------------------------
- Implemented `export_meals` view that exports historical Meal records for the
  current user (temporarily using the first UserProfile until auth is wired).
- The view reads ?format= from the query string (csv or json, default csv).
- For JSON export:
  - We build a list of plain dicts for each meal (meal_id, total_calories,
    total_protein, total_carbohydrates, total_fat, date).
  - We wrap it with metadata: generated_at ISO timestamp and record_count.
  - We return JsonResponse(data, json_dumps_params={"indent": 2}) and set
    Content-Disposition to attachment; filename="my_meals_YYYY-MM-DD_HH-MM.json".
- For CSV export:
  - We return HttpResponse(content_type="text/csv") and set Content-Disposition
    with a timestamped "my_meals_YYYY-MM-DD_HH-MM.csv" filename.
  - We write one header row followed by one row per Meal using csv.writer.
- These exports are linked from the reports UI as "Download CSV" / "Download JSON"
  so users can download their historical data in both formats.


SECTION 4: Reports Page & Aggregated Summaries
-----------------------------------------------
- Implemented `MealReportsView` (class-based View) that aggregates Meal data for
  a (temporary) current user and returns a structured JSON report.
- The view supports optional start/end date filters (YYYY-MM-DD) and computes:
  - total_count of meals,
  - category_stats: meals.values("category").annotate(num=Count("pk")),
  - macro totals via aggregate(Sum(...)) for calories, protein, carbs, and fat.
- From these, we derive labels/values arrays for macros and categories, which are
  used to generate a Matplotlib figure (two side-by-side pie charts) rendered to
  a base64 PNG string.
- The JsonResponse payload includes user_info, statistics (totals + grouped
  summaries), and chart_base64 (data:image/png;base64,...) so the frontend can
  show a visual report alongside totals.

SECTION 6: Creating APIs
--------------------------
API planning:
Since our project uses React, an API would be the primary way to connect the backend and frontend to send data; therefore,
every piece of data that we want to display in the frontend needs to be passed via an API.

Data served:
- /api/halls/        — dining hall names + locations
- /api/dishes/       — dish catalog, filterable by name (?search=)
- /api/dishes/<id>   — single dish detail
- /api/dish-stats/   — aggregation: total counts + grouped summaries
- /api/profiles/     — user profiles
- /api/meals/        — meal history
- /api/aimeals/      — AI meal search (GET fuzzy / POST exact)
- /api/dishes-manage/ — GET list / POST create dish

HttpResponse vs JsonResponse :
- dining_hall_view: HttpResponse(json.dumps(data), content_type=
  "application/json")
- dish_list_view: JsonResponse(data, safe=False)
- If HttpResponse omits content_type, Django defaults to text/html and
  the browser renders raw text. JsonResponse prevents this mistake.

Filtering via query parameters:
- /api/dishes/?search=chicken — icontains filter on dish_name.
- /api/aimeals/?dishes=Rice,Chicken — splits comma param, chains Q()
  with relationship spanning (items__dish__dish_name__icontains).
